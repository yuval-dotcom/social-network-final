# StudyCircle - נקודות להסבר בהגנה

## ארכיטקטורה

הפרויקט מחולק לשרת ולקוח:

- `server`: Node.js + Express.
- `client`: React Web רגיל בגרסת 19.2.x עם Vite.
- `MongoDB Atlas`: מסד הנתונים.

בשרת יש הפרדה לפי MVC:

- Model: הגדרת מבנה ולידציה של `User`, `Group`, `Post`.
- Controller: טיפול בבקשות HTTP.
- Routes: חיבור כתובות API לפונקציות controller.
- Repository: עבודה מול MongoDB native driver.

## שלושת המודלים הרשמיים

1. `User`
   - יצירה דרך מסך ההרשמה וגם דרך מסך `ניהול משתמשים`.
   - רשימה, חיפוש, עדכון ומחיקה דרך מסך משתמשים.
   - בא לידי ביטוי בפיד דרך המשתמש המחובר, החברים שלו, והקבוצות שאליהן הוא שייך.

2. `Group`
   - יצירה, רשימה, חיפוש, עדכון ומחיקה דרך מסך קבוצות.
   - מנהל קבוצה יכול לאשר משתמשים.
   - בא לידי ביטוי בפיד כקהילת לימוד שבתוכה מפרסמים פוסטים.

3. `Post`
   - יצירה, רשימה, חיפוש, עדכון ומחיקה דרך מסך פוסטים.
   - משתמש יכול לערוך ולמחוק רק פוסטים שלו.
   - בא לידי ביטוי בפיד כיחידת התוכן המרכזית: שאלה, סיכום, תגיות או וידאו.

משפט קצר להגנה:

```text
User מפרסם Post בתוך Group. הפיד לא שולף את כל הפוסטים בעולם, אלא מסנן לפי חברים וקבוצות שהמשתמש רשאי לראות.
```

## Auth והרשאות

- סיסמאות נשמרות כ-hash באמצעות `bcrypt`.
- אחרי התחברות השרת מחזיר JWT.
- הלקוח שומר את ה-JWT ושולח אותו בכל בקשה דרך `Authorization` header.
- השרת בודק את ה-token ב-middleware.

## מסך התחברות נפרד

- לפני התחברות מוצג מסך Auth עצמאי, בדומה למוצרי רשת חברתית כמו Facebook או Instagram.
- אחרי התחברות React מחליף מצב ומציג את הפיד הראשי של הרשת החברתית.
- פרטי ה-session נשמרים בצד לקוח: JWT עבור בקשות API ופרטי משתמש לתצוגה אחרי רענון.
- כפתור התנתקות מנקה את ה-token ואת המשתמש השמור ומחזיר למסך הכניסה.
- ההפרדה משפרת UX וגם מדגישה את דרישת ההרשאות: משתמש לא נכנס למסכי המערכת לפני login.

## פיד ראשי ומסך ניהול

- המסך הראשי אחרי התחברות הוא פיד חברתי: פוסטים מקבוצות וחברים שהמשתמש רשאי לראות.
- פרסום פוסט חדש מתבצע ישירות מהפיד, והפוסט מופיע מיד במסך.
- בצד הפיד יש כרטיס פרופיל שמראה את המשתמש המחובר, מספר הפוסטים בפיד, קבוצות וכותבים פעילים.
- מסך `פרופיל` מציג חוויית משתמש רגילה שמחברת בין שלושת המודלים: פרטי `User`, קבוצות `Group` שבהן המשתמש חבר, ופוסטים אחרונים מסוג `Post`.
- כרטיס פוסט מציג משתמש, קבוצה, תאריך, תוכן, תגיות ופעולת שמירה מקומית כדי לתת תחושה של רשת חברתית.
- מסכי ה-CRUD של `User`, `Group`, `Post` לא מוצגים במסך הראשי כדי שהאפליקציה תרגיש כמו מוצר חברתי אמיתי.
- ה-CRUD עדיין עומד בדרישת המרצה כי הוא זמין דרך כפתור הניווט `ניהול`, עם יצירה/רשימה/חיפוש/עדכון/מחיקה דרך UI.
- במסך `ניהול משתמשים` יש גם טופס יצירת משתמש, כך שגם מודל `User` מכסה CRUD מלא דרך UI ולא רק דרך הרשמה.
- במסך `ניהול` יש גם "מפת מודלים" שמסבירה בצורה חזותית איך שלושת המודלים מתחברים.

## קומפוננטות React

- React כאן הוא React Web רגיל, לא React Native.
- שמרנו את React פשוט ועדכני: קומפוננטות פונקציונליות, `useState`, `useEffect`, props, וללא ספריית UI כבדה.
- כשיש ערכים שמחושבים מתוך state, כמו רשימת קטגוריות או סיכום פוסטים, הם מחושבים בצורה ישירה בתוך הקומפוננטה. לא השתמשנו ב-hooks מתקדמים שלא צריך, כדי שכל שורה תהיה קלה להסבר.
- Vite הוא כלי הרצה ובנייה ל-React, לא טכנולוגיית מוצר נפרדת.
- הקומפוננטות מחולקות לפי אחריות אחת, כמו בדוגמת מכונית: מסך הוא המכונית, וחלקי UI קטנים הם גלגל/הגה/מנוע.
- מסך הכניסה מופרד ל-`AuthPanel` שמנהל state ושליחה לשרת, ולחלקי UI: `AuthTopbar`, `AuthStory`, `AuthModeSwitch`, `AuthForm`.
- `FeedPanel` אחראי לטעינת הפיד וחיבור הנתונים, אבל הטופס, כרטיס הפוסט וה-sidebar מופרדים ל-`FeedComposer`, `FeedPostCard`, `FeedSidebar`.
- חלקים קטנים נוספים של הפיד מופרדים ל-`FeedProfileCard` ו-`FeedPostActions`.
- מסך הפרופיל מופרד ל-`ProfilePanel`, `ProfileHeader`, `ProfileStats`, `ProfileFriends`, `ProfileGroups`, `ProfileRecentPosts`.
- מסך ניהול המשתמשים מופרד ל-`UsersPanel` שמחזיק state ופעולות, ולרכיבי תצוגה קטנים: `UserCreateForm`, `UserSearchForm`, `UserEditForm`, `UserResultCard`.
- מסך ניהול הקבוצות מופרד באותה צורה: `GroupsPanel` מחזיק state ופעולות, ו-`GroupManagementForm`, `GroupManagementSearchForm`, `GroupManagementResultCard` מציגים את החלקים.
- מסך ניהול הפוסטים מופרד באותה צורה: `PostsPanel` מחזיק state ופעולות, ו-`PostManagementForm`, `PostManagementSearchForm`, `PostManagementResultCard` מציגים את החלקים.
- הסבר המודלים במסך הניהול מופרד ל-`ModelMap`.
- `GroupDiscoveryPanel` אחראי לטעינת קבוצות וחיבור הפעולות, אבל החיפוש, כרטיס הקבוצה ופרטי הקבוצה מופרדים ל-`GroupSearchBar`, `GroupCard`, `GroupDetailPanel`.
- `ChatPanel` אחראי לחיבור Socket.io ולניהול הודעות, ו-`ChatSidebar`, `ChatRoomControls`, `ChatTranscript`, `ChatComposer` מציגים את חלקי הצ'אט.
- `StatsPanel` טוען נתונים מהשרת, ו-`D3BarChart` הוא הרכיב שמצייר את ה-SVG באמצעות D3.
- `MediaPanel` מחזיק ערכי טופס פשוטים, ו-`CanvasBadge` ו-`VideoPreview` מממשים בנפרד את דרישות Canvas ו-Video.
- רכיבים קטנים משותפים כמו `Avatar` ו-`TagChip` נמצאים בתיקיית `shared`.
- ההפרדה מקלה על בדיקות, שינוי עיצוב, והסבר בהגנה.

## מסך גילוי קבוצות

- כפתור `קבוצות` מציג מסך משתמש רגיל לגילוי קבוצות לימוד.
- המשתמש יכול לחפש לפי מילת חיפוש, קטגוריה ופרטיות.
- הצטרפות לקבוצה ציבורית מוסיפה את המשתמש כחבר.
- בקשת הצטרפות לקבוצה פרטית עוברת לסטטוס המתנה לאישור מנהל קבוצה.
- זה נפרד ממסך `ניהול קבוצות`, שנשאר מסך CRUD מלא לצורך דרישות הפרויקט.

## מסך הפוסטים שלי

- כפתור `הפוסטים שלי` מציג רק את הפוסטים שהמשתמש המחובר פרסם.
- המסך משתמש ב-endpoint `/api/posts/mine`.
- המשתמש יכול לערוך ולמחוק פוסטים שלו דרך מסך רגיל, לא רק דרך CRUD.
- השרת עדיין אוכף הרשאות: משתמש לא יכול לערוך או למחוק פוסט של משתמש אחר.
- זה מממש את דרישת "הפוסטים שלי" כחוויית משתמש עצמאית.

## jQuery Ajax

למרות שה-UI כתוב ב-React, כל הקריאות לשרת עוברות דרך `$.ajax` בקובץ אחד:

```text
client/src/api/http.js
```

זה עונה לדרישת שימוש ב-jQuery Ajax.

## Socket.io Chat

- הלקוח שולח `chat:send`.
- השרת שומר את ההודעה ב-MongoDB.
- השרת מפיץ `chat:message` לכל מי שנמצא באותו room.
